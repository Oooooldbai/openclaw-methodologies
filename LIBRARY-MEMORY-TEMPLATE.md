# 图书馆记忆法 (Library Memory System)

> 建立索引 → 按需加载 → 减少token消耗 | v1.1

---

## 核心思想

**像图书馆一样管理 AI Agent 的记忆。**

| 图书馆 | AI Agent |
|--------|---------|
| 不会把所有书放在桌上 | 不会加载所有记忆到上下文 |
| 建立索引系统 | 建立关键词映射表 |
| 按需取书，读完放回 | 按需加载记忆，用完释放 |

---

## 解决的问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| **启动慢** | 每次加载所有文件 | 只加载轻量索引（<500 tokens） |
| **信息丢失** | 新会话不记得配置 | 关键配置永久存储 |
| **无法按需加载** | 不知道信息在哪个文件 | 关键词映射表 |
| **配置泄露** | token散落各文件 | 四层分离存储 |
| **日志堆积** | 每日文件越来越多 | 生命周期管理 |

---

## 四层配置体系（v1.1）

```
┌─────────────────────────────────────────────┐
│  索引层 (index.md)                           │
│  关键词 → 文件映射，<500 tokens              │
│  职责：启动时第一个读，知道去哪找              │
└──────────────────┬──────────────────────────┘
                   ↓
┌─────────────────────────────────────────────┐
│  核心层 (MEMORY.md)                          │
│  配置摘要，不含敏感值                         │
│  职责：永久存储"有什么配置"                    │
└──────────────────┬──────────────────────────┘
                   ↓
┌─────────────────────────────────────────────┐
│  敏感层 (config.md)                          │
│  实际 token 值，已 gitignore                  │
│  职责：安全存储"具体值是什么"                  │
└──────────────────┬──────────────────────────┘
                   ↓
┌─────────────────────────────────────────────┐
│  工具层 (TOOLS.md)                           │
│  使用说明，引用 config.md                    │
│  职责：记录"怎么用"                           │
└──────────────────┬──────────────────────────┘
                   ↓
┌─────────────────────────────────────────────┐
│  知识层 (topics/*.md)                        │
│  按需加载的专题知识                           │
│  职责：存储详细内容                           │
└─────────────────────────────────────────────┘
```

### 四层职责

| 层 | 文件 | 存什么 | 何时读 |
|---|------|--------|--------|
| **索引** | `memory/index.md` | 关键词→文件映射 | 启动必读 |
| **核心** | `MEMORY.md` | 配置摘要（"有XX Key，用途YY"） | 按索引加载 |
| **敏感** | `config.md` | 实际 token 值 | 按需加载 |
| **工具** | `TOOLS.md` | 工具使用说明 | 使用工具时 |
| **知识** | `topics/*.md` | 专题详细内容 | 对话触发时 |

### 配置安全规则

> ⚠️ 任何新配置必须四层同步更新。

```markdown
# 示例：新增一个 API Key

## 索引层 (index.md)
| "API名称" / "用途关键词" | MEMORY.md → config.md | 说明 |

## 核心层 (MEMORY.md)
### API配置
- API名称 Key：用于XXX（见 config.md）

## 敏感层 (config.md)
| 服务 | Key | 备注 |
|------|-----|------|
| API名称 | sk-xxxxxxxxx | 用途说明 |

## 工具层 (TOOLS.md)
- API名称 Key：见 config.md
- 用途：XXX
- 使用方法：XXX
```

## 目录结构

```
workspace/
├── MEMORY.md              ← 核心层（配置摘要 + 项目结论）
├── config.md              ← 敏感层（实际 token，gitignore）
├── TOOLS.md               ← 工具层（使用说明）
├── memory/
│   ├── index.md           ← 索引层（关键词映射，启动必读）
│   ├── YYYY-MM-DD.md      ← 每日日志（最近2天）
│   ├── archive/           ← 归档日志（7天+）
│   └── topics/            ← 知识专题（按需加载）
│       ├── topic-a.md
│       └── topic-b.md
└── AGENTS.md              ← Agent 行为规则
```

## 启动流程

```markdown
## Session Startup

1. Read identity files (SOUL.md, USER.md, etc.)
2. Read `memory/index.md` — 索引层，知道有什么
3. Read `memory/YYYY-MM-DD.md` (昨天+今天) — 最近上下文
4. **按需读 `MEMORY.md`** — 核心层，知道配置在哪
5. **需要实际值时读 `config.md`** — 敏感层
6. **不加载 topics/*.md** — 对话触发时再加载
```

---

## 对话中按需加载

```markdown
## 规则
- 不一次性加载所有 topics，根据对话关键词按需读取
- 关键配置从 MEMORY.md 读取，不问用户
- 需要具体 token 值时，从 config.md 读取
```

关键词映射示例：

| 关键词 | 加载文件 | 说明 |
|--------|----------|------|
| "API名" / "用途关键词" | `MEMORY.md` → `config.md` | API配置 |
| "项目A" / "主题X" | `topics/project-a.md` | 专题知识 |
| "方法论" / "记忆结构" | `AGENTS.md` + `index.md` | 系统管理 |

## 每日日志生命周期

```
Day 0          Day 1-7              Day 7+              Day 30+
写入 → 启动时读取提供上下文 → 蒸馏提炼有价值内容 → 归档/清理
```

| 阶段 | 时机 | 动作 | 规则 |
|------|------|------|------|
| **写入** | 当天随时 | 重要事件、决策、配置变更 | 全量记录，不过滤 |
| **读取** | 每次启动 | 读最近2天日志 | 提供上下文 |
| **蒸馏** | 几天后 | 有价值内容 → topics/ 或 MEMORY.md | AI+人工 |
| **归档** | 7天后 | 移到 memory/archive/ | 避免堆积 |
| **清理** | 30天后 | 删除无价值的日志 | 节省空间 |

### 日志与配置的区别

| 维度 | 配置信息 | 每日日志 |
|------|---------|---------|
| **生命周期** | 永久保存 | 有生命周期 |
| **存储位置** | 四层结构 | 日志文件 |
| **更新频率** | 变更时同步四层 | 当天随时写入 |
| **最终去向** | 长期驻留 | 蒸馏→归档→清理 |

## 重要性同步触发条件

### 为什么需要？

按需加载机制依赖 `index.md` 的关键词映射。如果重要信息更新后没同步到索引，下次就检索不到。

### 触发条件（任一满足即执行）

| 触发源 | 触发词/条件 |
|--------|------------|
| 用户反馈 | "太好了" / "解决了" / 明确认同 |
| **Agent自己的判断** ⚠️ | "太好了" / "解决了" / "已完成" / "这个很重要" |
| 内容类型 | 新概念/名单/分类体系/核心配置 |

> ⚠️ **关键洞察**：Agent说"太好了"时，用户会默认已识别重要性。如果Agent不执行同步，信息就断层。

### 同步动作（三步）

```
1. 更新 MEMORY.md（核心层）— 摘要式记录
2. 更新 memory/index.md（索引层）— 添加关键词映射
3. 在 daily notes 标记⭐ — 方便回顾
```

## Token 节省效果

| 场景 | 传统方式 | 图书馆记忆法 | 节省 |
|------|---------|-------------|------|
| 启动 | 10000+ tokens | ~500 tokens | 95% |
| 知识查询 | 逐个搜索所有文件 | 按索引直接加载 | 80% |
| 配置查询 | 可能丢失或重复问 | 固定路径读取 | 100% |

## 常见错误

| ❌ 错误 | ✅ 正确 |
|---------|---------|
| 每次问用户配置信息 | 永久存在 MEMORY.md |
| 启动加载所有文件 | 只加载索引，按需读取 |
| 配置散落各文件 | 四层统一管理 |
| 敏感 token 存在普通文件 | 只存 config.md（gitignore） |
| 重要讨论后不更新索引 | 触发同步条件，三步更新 |
| 日志无限堆积 | 蒸馏→归档→清理 |

## 与其他方法论的关系

- **图书馆记忆法**：解决"记忆问题"（启动/存储/检索）
- **消防队模式**：解决"执行问题"（任务拆解/检查点/恢复）
- **任务闭环**：解决"确认问题"（确认→固化→验证）

三者结合 = 完整的 **记忆 + 执行 + 质量保障** 体系。
