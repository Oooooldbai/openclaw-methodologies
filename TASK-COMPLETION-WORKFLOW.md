# 任务闭环管理

> 确认→固化→验证，缺一不可

---

## 核心理念

**确认 ≠ 执行，确认 → 固化 → 验证**

配置/规则/格式确认时，必须同时更新：
1. **执行配置**（cron/脚本/参数）
2. **检查点文件**（checkpoints/confirmation/*.json）
3. **长期记忆**（MEMORY.md关键配置区）
4. **验证测试**（立即执行一次）

---

## 检查清单

- [ ] **执行配置**更新了吗？
- [ ] **检查点**创建了吗？
- [ ] **MEMORY.md**更新了吗？
- [ ] **验证测试**执行了吗？
- [ ] **任务追踪**记录了吗？

**任一没做 = 没有真正固化**

---

## 消防队拆解（<5分钟子任务）

| 阶段 | 任务 | 检查点文件 |
|------|------|-----------|
| 确认 | 识别影响范围 | `{id}-impact.json` |
| 固化 | 更新配置+记忆 | `{id}-config.json` |
| 验证 | 测试+验证 | `{id}-verify.json` |

---

## 检查点格式

```json
{
  "task_id": "xxx",
  "phases": {
    "confirmation": {"completed": true, "timestamp": "..."},
    "solidification": {"completed": true, "config_updated": [...]},
    "verification": {"completed": true, "test_result": "pass"}
  },
  "files_updated": ["cron", "MEMORY.md"]
}
```

---

## 触发条件

- 用户说："确认"/"就这么定了"
- 讨论超过3轮
- 涉及：配置变更/格式标准/自动化任务

---

## 实战流程

```bash
# 1. 确认
# 用户确认内容 → 创建impact检查点

# 2. 固化
# 更新cron/脚本 + MEMORY.md + 关键词映射
# 创建config检查点

# 3. 验证
cron run <job_id>  # 立即测试
# 检查输出 → 创建verify检查点

# 4. 归档
# 合并检查点 + 更新task-tracking.md
# MEMORY.md标记 ⭐
```

---

## 与现有体系整合

**图书馆记忆法**：检查清单按需加载，关键词"任务完成"/"确认固化"/"验证"

**消防队模式**：拆解<5分钟子任务，独立检查点，支持中断恢复

---

## 总结

**没有固化的确认 = 口头承诺**  
**没有验证的固化 = 自我安慰**  
**没有记忆的验证 = 下次再犯**
