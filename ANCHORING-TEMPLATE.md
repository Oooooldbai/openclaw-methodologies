# 锚定机制（Anchoring Mechanism）

> 识别重要时刻 → 立即锁定 → 不可丢失
> 不依赖"想起来"，依赖系统强制触发。

---

## 核心问题

Agent行为教训、重要决策、关键洞察——这些信息经常丢失。原因：
1. **识别失败** — 发生了但没意识到重要性
2. **固化失败** — 意识到了但只写了日志（日志≠固化）
3. **加载失败** — 固化了但下次启动没读到

锚定机制解决的是**触发识别**和**立即固化**。

---

## 触发条件（三类锚定）

### A类：强制锚定（做完即锁，不等"确认"）

| 触发场景 | 锚定内容 | 写入位置 |
|---------|---------|---------|
| 建立新协作机制 | 机制名、通信方式、文件位置 | SOUL.md 或 AGENTS.md |
| 新增/变更关键配置 | 配置项、位置、用途 | config.md 或 config.local.md |
| 修改规则文件 | 改了什么、为什么改 | FIREFIGHTER-MODE.md 反面案例区 |
| 用户/自己说"这个很重要" | 内容+上下文 | MEMORY.md + index.md |
| **犯了错** | 过程+根因+正确做法 | SOUL.md Red Lines + FIREFIGHTER-MODE.md 反面案例 |
| 建立新文件结构/目录 | 结构、用途、命名规则 | MEMORY.md + config.md |
| **知识库入库** | 文件路径+关键词映射 | memory/index.md（关键词映射）+ 日志（入库记录） |
| **收到有价值的内容**（文件/长段文字/原文/截图OCR结果） | 存到知识库对应项目目录 | Laobai-Second-Brain/50-Projects/ 或 topics/ |

### B类：过程自检（触发时强制执行）

| 触发场景 | 动作 |
|---------|------|
| 用户提到你没读到的上下文（如"昨晚发的帖有回复"） | 立刻停止回复 → 读日志/上下文 → 再回复 |
| 图片/数据看不清 | 立刻OCR/工具验证 → 不确定就说不确定 |
| 不确定的内容 | 说"我不确定"或用工具验证，不猜 |

**核心：先补再回，不是先回再补。**

---

### C类：确认锚定（用户明确确认后）

| 触发词 | 动作 |
|--------|------|
| "确认"/"就这么定了"/"以后按这个来" | 立即执行确认→固化→验证闭环 |
| 讨论同一话题超过3轮 | 暂停讨论，先固化已有结论 |
| "太好了"/"解决了" | 立即同步到 MEMORY.md + index.md |

### D类：巡检锚定（定期扫描发现）

- HEARTBEAT 扫描 `pending-anchors.json`，发现未处理的锚定请求
- 每周记忆架构自检时，检查是否有遗漏的重要内容

---

## 执行动作（三步锁定）

```
Step 1: 识别 → 检测到触发条件
Step 2: 判断 → 这个锚定该写进哪个系统文件？
Step 3: 写入 → 立即写入，不等"收工"，不等"确认"
```

**判断写入位置的优先级：**

| 信息类型 | 写入位置 | 原因 |
|---------|---------|------|
| 犯错/行为教训 | SOUL.md Red Lines | 每次启动必看，零容忍 |
| 犯错过程复盘 | FIREFIGHTER-MODE.md 反面案例 | 启动时强制面对 |
| 执行规则/流程变更 | AGENTS.md | 启动流程的一部分 |
| 配置信息 | config.md / config.local.md | 单一来源 |
| 结论/决策 | MEMORY.md + index.md | 长期记忆 |
| 详细知识 | topics/*.md | 按需加载 |

## pending-anchors.json（待处理锚定队列）

当识别到重要时刻但无法立即固化时（如正在执行复杂任务），写入待处理队列：

```json
{
  "pending_anchors": [
    {
      "id": "anchor-001",
      "detected_at": "2026-04-23T09:00:00+08:00",
      "trigger_type": "A",
      "trigger_scene": "犯错",
      "summary": "图片识别欺骗：编造内容，未使用OCR",
      "target_files": ["SOUL.md", "FIREFIGHTER-MODE.md"],
      "status": "pending",
      "resolved_at": null
    }
  ],
  "last_checked": "2026-04-23T09:00:00+08:00"
}
```

**HEARTBEAT 扫描逻辑：**
- 发现 status=pending → 立即处理
- 超过24小时未处理 → 标记为 overdue，下次HEARTBEAT优先处理
- 处理完成后 → status=done

## 与其他两个机制的关系

```
图书馆记忆法（存储层）  →  存什么、怎么存、怎么读
消防队模式（执行层）    →  任务怎么拆、怎么并行、怎么恢复
锚定机制（触发层）      →  什么时候该固化、怎么固化
```

- 锚定机制决定**什么时候**要固化
- 图书馆记忆法决定**存哪里**
- 消防队模式决定**执行过程中的检查点怎么管**

三者独立但互补，共同构成方法论三足鼎立。

## 过程自检（触发式，不靠自觉）

事后固化是兜底，事中触发是防线。

**但"每次回复前自检"在当前架构下无法自动触发——没有hook机制。**

所以自检不靠自觉，靠触发：

1. **启动时** — AGENTS.md Session Startup 第6步：检查待处理上下文，标记未完成项
2. **用户提到未知上下文时** — 立刻停止回复，先读日志/文件，再回复
3. **图片/数据不确定时** — 立刻用工具验证，不猜

这三条写进 SOUL.md（回复前硬规则）+ AGENTS.md（启动流程），每次加载强制可见。

**不依赖"想起来"，依赖触发条件。**

---

## 反模式（不要做）

| 反模式 | 为什么错 | 正确做法 |
|--------|---------|---------|
| "记下来"然后继续 | 日志≠固化，下次启动读不到 | 立即写入系统文件 |
| "等收工再同步" | 收工会忘 | 识别到立即锁 |
| "我大概知道" | 模型重启后丢失 | 写进启动必加载的文件 |
| 只写日志不写系统 | 日志有生命周期会被清理 | 写进SOUL/AGENTS/FIRE |
| 犯错后只道歉不改系统 | 下次犯同样的错 | 教训写进Red Lines和反面案例 |
| "做不到"没试就说 | 有工具/权限但不尝试 | 先试再说做不到 |
